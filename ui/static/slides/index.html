#1 — fix-task (LLM)

Το LLM που καθαρίζει το raw ASR. Ήδη production. Εκεί έτρεξα το glossary A/B.

#2 — ASR fine-tuning dataset

Ζεύγη (audio, σωστό κείμενο) για να εκπαιδεύσουμε το ακουστικό μοντέλο. Αυτό είναι το κυρίως θέμα.

Το #1 τροφοδοτεί το #2 ως triage: μας λέει ποια λάθη τα πιάνει ήδη το LLM (χαμηλή αξία για το ακουστικό) vs ποια είναι γνήσια ακουστικά (υψηλή αξία).

Στόχος-μετρική: HIR = πόσα utterances χρειάζονται ανθρώπινη διόρθωση. Σήμερα ~28%. Θέλουμε να πέσει.

Track 2 — τι έδειξε το CPU auto-research

  • Το καλό: fine-tune σε ~17 λεπτά corrections κάνει την καθημερινή ομιλία σαφώς καλύτερη, σε κάθε run. val_reg: 0.674 → ~0.43 (~24 μονάδες). Μαθαίνει το domain (λεξιλόγιο, ονόματα), δεν αποστηθίζει.
  • Το πρόβλημα: τα δύσκολα corrected cases (val_corr) δεν κουνιούνται — μένουν ~0.61. Και η μέτρηση είναι θόρυβος: ίδιο config με άλλο seed κουνιέται 0.062 — όσο και η διαφορά ανάμεσα σε όλα τα configs (0.064).
  • Ένας φόβος δεν βγήκε αληθινός: το να εκπαιδεύεις μόνο σε λάθη δεν χάλασε την κανονική ομιλία. Έγινε καλύτερη.
Άρα: με μόνο 56 val κλιπ δεν μπορώ να συγκρίνω data recipes ακόμα. Πρώτη δουλειά: φτιάξε το validation set. Αυτό είναι το πραγματικό blocker, όχι η επιλογή μίγματος.

Kaggle — large-v3 dry-run

Το έτρεξα στο Kaggle (T4×2, headless) σε ~1.9k εγκεκριμένα corrections (~1.8h ήχου).

  • Τι ήθελα να δείξω: ότι το pipeline δουλεύει end-to-end στο κανονικό μοντέλο — fetch → κόψιμο κλιπ → LoRA → WER στο held-out. Αυτό δούλεψε.
  • Memory-safe build: κάθε meeting mp3 αποκωδικοποιείται μία φορά και ελευθερώνεται (παλιά έσκαγε σε OOM).
Τι ΔΕΝ είναι: δεν είναι απόδειξη ότι βελτιώνει το production. 1.8h είναι λίγα → αναμένεται overfit. Το διαβάζω ως «τρέχει και μαθαίνει κάτι», όχι ως «κερδίσαμε».

Kaggle — τι μπορεί να είναι λάθος (το διορθώνω)

  • Πολύ μικρό & θορυβώδες validation. Τα νούμερα στα δύσκολα cases δεν είναι αξιόπιστα. → Φτιάχνω val από όλο το human-corrected pool στις held-out πόλεις (~9.9k αντί 56).
  • Leakage στο val_reg. Τραβιέται από τα ίδια meetings με το val_corr — ίδιες αίθουσες/ομιλητές. Η βελτίωση είναι ενθαρρυντική αλλά μάλλον πιο εύκολη από αληθινό generalization. → Ξεχωριστό regression set από meetings που δεν έδωσαν corrections.
  • Ένα μόνο seed. → ≥3 seeds, mean/range, bootstrap CIs ανά meeting.
  • Λίγα δεδομένα (1.8h). → προς 30–50h για «πρώτο χρήσιμο checkpoint» πριν διαβάσω mixture effects ως αληθινά.
  • Το learning rate / steps από CPU run δεν μεταφέρονται — re-tune στο GPU.

LLM vs fine-tune — το πείραμα

Ίδιος held-out ήχος, 4 transcripts, ίδιο scoring. 114 κατηγοριοποιημένα κλιπ από πόλεις που δεν είδε το μοντέλο.

ΣτάδιοWER (normalized)
A — raw ASR0.715
B — ASR + LLM0.578
C — fine-tuned ASR0.613
D — fine-tuned + LLM0.478

Το fine-tune ρίχνει ~0.10. Το LLM ρίχνει ~0.14. Μαζί → ~το άθροισμα. Το LLM ρίχνει ~0.135 είτε έγινε fine-tune είτε όχι (D−C ≈ B−A), και το αντίστροφο. Διορθώνουν διαφορετικά λάθη — κράτα και τα δύο.

Προσοχή: 114 κλιπ, 1 seed. Το λέω ως κατεύθυνση για το επόμενο πείραμα, όχι «το αποδείξαμε».

Οι υπόλοιπες κατηγορίες είναι directional (μικρά n) — υπόνοιες, όχι ευρήματα.

Πρόταση #1 — εστίαση στα φωνητικά, λιγότερη στίξη

Για το επόμενο dataset, να δώσουμε βάρος σε φωνητικά / ακουστικά λάθη — εκεί φαίνεται να βοηθά το fine-tune (phonetic, homophones, word boundaries, ονόματα για να πλησιάσει ακουστικά).

Να μειώσουμε ή να πετάξουμε τα punctuation-only edits — αυτά τα πιάνει το LLM ούτως ή άλλως, και το fine-tune δεν προσθέτει τίποτα εκεί. Μπορώ να βάλω και φίλτρο που τα κόβει εντελώς.

Όχι μόνο στίξη: και τόνοι, αριθμοί/ημερομηνίες, γραμματική — όλα στο LLM. Κρατάμε τον ~50% κορμό χωρίς-edit (γι' αυτό δεν χάλασε η κανονική ομιλία).

Πρόταση #2 — ποια edits αξίζουν στο ακουστικό

edit classnLLM fix rate
task_only (μόνο το LLM)50660.7%
task_then_user (LLM + άνθρωπος)23641.9%
user_only (μόνο άνθρωπος)25847.4%
  • Τα task_only είναι αυτά που το LLM ήδη διορθώνει πιο εύκολα. Πρόταση: χαμηλότερη προτεραιότητα στο ακουστικό — εμπιστευόμαστε το task να συνεχίσει να τα πιάνει.
  • Πιο ενδιαφέροντα: task_then_user και user_only — εκεί το αυτόματο pipeline δεν έφτασε, τα γνήσια δύσκολα.
Όχι «τα πετάμε όλα»: το 60.7% σημαίνει ότι ~39% μένουν ακόμα άλυτα. Κρατάμε τα task_only edits που έχουν πραγματική ακουστική/φωνητική δυσκολία· τα υπόλοιπα απλώς down-weight.

Επόμενα βήματα — με τη σειρά

  1. Φτιάξε πρώτα το validation set. Είναι το πραγματικό blocker. Μεγαλύτερο val_corr + ξεχωριστό, χωρίς-leakage regression set. Στόχος: seed swing < 0.01.
  2. ≥3 seeds ανά config + bootstrap CIs by default. Τότε το ranking των recipes αποκτά νόημα.
  3. Χτίσε το dataset με τα παραπάνω βάρη: φωνητικά ↑, στίξη/task-only ↓.
  4. Τρέξε large-v3 + LoRA σε GPU με τη composition recipe, σύγκρινε με το baseline.

Σύνοψη

Fine-tune και LLM είναι συμπληρωματικά (preliminary). Το fine-tune αξίζει στα φωνητικά· η στίξη μένει στο LLM.
