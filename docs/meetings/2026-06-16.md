## What the benchmark is (and is not)

- It is a **provider comparison** on a frozen, randomly-seeded sample drawn across
  *all* meetings.
- It is **not** the finetune test set — its windows mix utterances from meetings
  that will be used for training (leakage). Kept as a provider scoreboard only.

## Decisions captured

→ all written into [../decisions/data.md](../decisions/data.md):

- **Cost is not a deciding factor** — optimise for transcription quality (Christos).
- **Test set = meetings/cities reviewed from 1 Jun 2026 onward** (temporal hold-out;
  new cities + new speakers; naturally uncontaminated since training won't touch
  current review work). A separate benchmark restricted to post-June meetings
  becomes the official finetune test set, re-run at the end to compare baseline →
  finetuned.
- **Train/Val split = pre-June data, seeded + automated, by whole meeting + speaker.**
  No human picking of speakers (would bias the set). Hold out entire jurisdictions
  (e.g. all speakers of a city) into the held-out set. Target ~30/70, both sets
  must have sufficient data.

## Prefetch bug — root cause + fix (done 2026-06-16)

Investigated systematically; the original "clean the index" idea turned out to be a
**minor lever**, not the fix. Two confirmed root causes (code-grounded):

- **(A) Request volume per hop.** `audioPool.setActive` warmed
  `neighborsAround(id, 10)` = up to ±10 ≈ 20 `<audio>` elements
  (`preload=auto` + seek) → ~20 CDN range requests **every** navigation hop.
  Fast/consecutive hops (404 auto-skip, fast key-nav) saturate the browser's
  ~6-connections-per-host limit → the visible clip queues behind warm fetches →
  the 700ms–2.6s / ">10s" latency Eliza saw. Same volume for the ±10 transcript
  context prefetch.
- **(B) Window vs nav-path mismatch.** Prefetch warmed **raw** seeded neighbours,
  but navigation advances by **filtered** adjacency (`nextUnreviewedId` /
  `autoSkipPrivate` jump past classified/private) → reviewer lands on a **cold**
  item ("not ready when it should be").

Why "clean the index" is only minor: eligibility is a **query-time, threshold-only**
filter — it knows nothing about public/private (private is discovered live via
404). The eligible corpus is 384 meetings / 15 cities, of which the 10 public
cities hold 379; restricting to them drops only **~5 meetings (~1.3%)** and can't
remove private meetings *inside* public cities nor individually-removed utterances.

**Fix (surgical, no re-upload):** new `queue.prefetchWindow(id, forward, back,
skipClassified)` — forward-biased, skip-aware, walk-to-fill, cached-only. Wired
into both prefetch effects with **FWD=6 / BACK=2** (down from ±10). Auto-skip path
left unchanged (still needed for the residual removed-utterance 404s).
Plan reviewed by Codex; unit-tested (`group-queue-prefetch-window.test.ts`);
182/182 unit tests + svelte-check clean. Files:
`ui/src/lib/client/group-queue.svelte.ts`,
`ui/src/routes/review/[utterance_id]/+page.svelte`.
